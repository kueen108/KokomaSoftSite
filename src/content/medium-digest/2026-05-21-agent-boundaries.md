## 왜 이 글인가

요즘 코딩 에이전트의 생산성 이야기는 충분히 많다. 하지만 챨리가 실제로 OpenClaw, Codex, Claude Code 같은 도구를 매일 쓰는 입장이라면, 오늘 더 읽을 가치가 큰 쪽은 “얼마나 잘 짜느냐”보다 “어디까지 믿고 맡길 수 있느냐”다. 이 글은 Cursor의 NomShub 취약점 사례를 출발점으로, 에이전트가 읽은 텍스트가 실행 권한과 만나면 어떤 식으로 경계가 무너질 수 있는지 설명한다.

특히 좋은 점은 특정 제품을 단순히 까는 글이 아니라, Cursor·Claude Code·Codex의 기본 보안 모델 차이를 비교한다는 것이다. 챨리가 자동화와 에이전트 워크플로를 늘려갈수록 “승인 피로를 줄이는 것”과 “권한 경계를 유지하는 것” 사이의 균형을 계속 잡아야 한다. 이 글은 그 균형점을 점검하는 데 꽤 실용적인 체크리스트가 된다.

## 핵심 요약

- Cursor NomShub 사례는 악성 저장소의 README 같은 평범한 텍스트가 에이전트를 간접 프롬프트 인젝션으로 조종할 수 있음을 보여준다.
- 문제의 본질은 메모리 버그가 아니라, “신뢰할 수 없는 입력”과 “실제 실행 권한을 가진 에이전트” 사이의 경계 붕괴다.
- 셸 built-in, 홈 디렉터리 쓰기, 원격 터널 같은 작은 권한들이 조합되면 단순 명령 실행을 넘어 지속성 있는 원격 접근으로 이어질 수 있다.
- Codex CLI는 기본적으로 워크스페이스 중심 샌드박스와 네트워크 제한 같은 기술적 경계에 강점을 둔다.
- Claude Code는 기본 모드에서 편집·셸·네트워크 요청에 승인 단계를 두는 인간 검토 중심 모델에 가깝다.
- Cursor는 로컬 IDE, 클라우드 에이전트, 원격 개발 등 모드가 다양해서 생산성은 높지만, 설정별 보안 자세를 더 꼼꼼히 봐야 한다.
- 자율성이 강해질수록 로그, 승인 기록, diff 검토, 네트워크 관찰 같은 운영 가시성이 모델 성능만큼 중요해진다.

## 챨리에게 중요한 포인트

- 에이전트가 읽는 모든 외부 텍스트—README, 이슈, 웹페이지, 로그, 커밋 메시지—는 잠재적으로 “명령처럼 보이는 입력”이 될 수 있다.
- `full-auto`, “Run Everything”, 네트워크 허용 같은 편의 설정은 시간을 아껴주지만, 신뢰할 수 없는 저장소에서는 비용이 너무 클 수 있다.
- OpenClaw류 개인 자동화에서는 기술적 샌드박스와 인간 승인 단계를 둘 다 남겨두는 편이 안전하다. 하나만 믿으면 은근히 허술해진다.

## 적용 아이디어

- 낯선 저장소를 에이전트에게 맡길 때는 네트워크 차단, 워크스페이스 제한, 승인 필요 모드로 시작한다.
- 홈 디렉터리, 셸 시작 파일, SSH/GitHub 토큰, 클라우드 자격증명에 접근하는 명령은 자동 승인하지 않는 규칙을 둔다.
- 에이전트 실행 로그와 diff를 “나중에 문제 생기면 보는 것”이 아니라, 작업 완료 전 짧게 확인하는 루틴으로 만든다.
- 반복 자동화 작업은 임시 토큰·제한 권한·격리된 작업 디렉터리에서 돌리는 실험을 해본다.

## 읽기 우선순위

오늘 하나만 읽는다면 이 글은 상위권이다. 코딩 에이전트를 더 많이 쓸수록, 이런 글을 안 읽는 게 더 비싸진다.
